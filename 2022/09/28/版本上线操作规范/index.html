    规范说明：

    此规范用于深圳创巧数据技术有限公司（下称：创巧数据）版本上线规范填写标准上线文档、上线流程、上线备份还原等上线操作规范。如有文档填写不标准一律要求重新填写上传，填写标准再上线。为了避免填写上线文档人员依赖审核人员当测试上线文档工具，同样问题超三次后不予说明退回重新填写。发版人员也必须按照上线流程规范自己安标准流程执行。

    一、上线文档SVN目录

    上线文档规范每次版本上线提交到SVN目录。SVN目录如下：

    ​ svn://192.168.2.183/warehouse_receipt/创巧动产数字化服务平台/08运维部署/

    例如：此次上线为v1.3.0版本

    ​ SVN目录为： svn://192.168.2.183/warehouse_receipt/创巧动产数字化服务平台/08运维部署/v1.3.0

    二、上线流程

    1、UAT环境上线测试

    ​ 生产环境上线：UAT环境作为上线前最后一次上线演练，整理之前上线的需要的步骤和上线的文件、服务应用等。

    2、上线评审

    ​ 联合开发、测试、架构和运维团队对上线的文档进行详细的评估及检查文档的遗漏问题、风险性等。

    3、上线审批

    ​ 待上线评审结束，需要对此次上线版本申请上线。待产品部门确定上线时间及发出上线通告、CTO确认，运维团队根据时间进行版本上线。

    4、上线前确认、上线操作人员核对上线文档

    ​ 确认各项目的上线文档上传完成、git分支合并发布环境完成。才可进入下一步操作

    ​ 核对上线文档是否有格式不对、描述不清楚、内容错误等问题。如有和上线文档标准不符要求退回重新填写上传。

    ​ 有新项目部署应提前做好jenkins自动构建。

    5、上线前备份

    ​ 上线操作前必先备份好所有平台相关数据库(包括 nacos)、上个版本的项目服务程序包。

    6、上线操作流程

    A、运维A更新配置及sql数据。

    B、运维B对更新的配置和SQL进行检查。

    C、运维A更新服务程序。

    因项目服务程序部署有依赖API关系，构建时需要对API本地依赖包清理。

    例如：清理blade-common本地包

    rm -rf /root/.m2/repository/org/springblade/blade-common/

    D、运维B对项目服务程序进行初步的检查及测试。

    E、上线完成报告：

    ​ 上线完后通知测试人员及开发主管上线完成。填写本次上线完成报告、记录部署中出现的问题。

    三、填写上线文档标准

    1、目录结构

    A、目录层级到项目目录结束，不要再建子目录了。

    ​ 需要上线的内容在版本发布和回滚计划中写清楚，描术不清楚应写到项目目录下的文件中。不要再建子目录增加文件，上线内容一多,子目录容易被忽略。

    SVN发版控制
    v1.1.0
        ...
        cangdan
            nacosV1.1.0.yaml
            cangdanV1.1.0.sql
            email_templatev1.1.0.tmpl
            cangdan版本发布和回滚计划_v1.1.xlsx
            ...
        bladex
            nacosV1.1.0.yaml
            bladexV1.1.0.sql
            bladex版本发布和回滚计划_v1.1.xlsx

    v1.2.0
         ...

    2、版本发布和回滚计划_vx.x.x填写

    A、版本发布excle应写清楚项目所有需要上线的内容

    ​ 版本发布excle是上线优先打开按内容操作的。应写全本项目需要上线的所有内容。另外具体步骤中写清楚步骤文件位置

    操作序号 操作内容 具体步骤 回滚计划
    1 nacos配置更新 转到 ./项目bladex/nacos_blade.yaml …
    B、具体步骤栏一项不能又填执行文件，又填执行命令。

    ​ 一个操作序号内要么分开填，要么全是执行文件或执行命令。两种执行方式非常容易漏掉内容。

    C、更新其它内容

    ​ 如果不能在excel 版本发布和回滚计划_vx.x.x 中写清楚，请在项目目录中新建文档写清楚需上线内容

    D、每个项目需单独填写一个版本发布和回滚计划excel

    ​ 如cangdan版本发布和回滚计划_v1.3.0.xlsx，里面的操作包含了仓单、cfca、文件（file-server）等多个项目的操作步骤。

    3、sql上线

    A、sql内容注意事项

    a、注释说明上线库名

    ​ 例如： – 本条sql文件上线到blade-{env}库； or 本条sql文件上线到cangdan-{env}库。

    b、在sql语句内容上一行，注释说明sql添加日期由谁增加及sql内容说明。方便后面有疑问的地方找添加者询问。

    ​ 例如：– 添加标仓告警处理时间。告警处理人、告警处理信息卢永成 添加日期2022-05-18

    c、上线sql后缀应 .sql 结尾，可导入更新到数据库。

    d、sql语句中不要写库名，因为mysql库在不同环境库名是不一样的。

    e、不能一个SQL文件中存在执行两个库的内容。

    f、UAT环境及生产环境不能有drop table语句，需要drop table的执行重命名。重命名规则为：tablename_bak$日期

    g、sql中出现乱码、sql执行出错一律退回重填。

    h、如需要复杂执行修改时，注释一定要描述清楚。执行人员看不清楚可退回重填或要求编写人员解释清楚。

    i、数据库版本上线固定执行更新的表格新增jenkins固定job.版本上线开启执行(执行上线人员操作)。

    ​ 例如： bladex库中blade-menu表。

    B、上线SQL模板
    vim cangdanV1.1.0.sql

    -- 本条sql文件上线到cangdan-{env}库
-- 徐洪笛，2022.6.1
-- 仓单质押表增加字段
ALTER TABLE t_cangdan_pledge ADD `cangdan_num` varchar(32) CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci DEFAULT NULL COMMENT '仓单数量';
ALTER TABLE t_cangdan_pledge_add ADD `cangdan_num` varchar(32) CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci DEFAULT NULL COMMENT '仓单数量';

    4、nacos配置更新

    A、nacos内容注意事项，未符合以下标准将退回重新填写。

    a、nacos需要增加或更改的内容写在一个文件中，并且要求使用yaml格式填写，能避免格式问题。文件名应为 nacos_{配置文件名}.yaml

    ​ 例如：nacos_blade-system-[env].yaml

    ​ 之前线上又有 update_nacos.txt 又有 xxx项目配置重构说明 .md ，两个文件内容也不一致，运维上线时难免漏掉。

    b、nacos内容填写不能加入tab键及其他符号，yaml是以双空格为等级划分符。

    ​ 推荐一个 https://www.toyaml.com/ 网站可以检测yaml格式。

    c、配置文件内容中：如果key对应的value每个环境不一致可标识{type/date.env}，其他需要修改的内容也需要使用大括号标识出来。让上线操作人员可区分。

    ​ 上线操作将大括号内容视为需要修改内容。其他内容视为固定内容将不做修改上线。如内容填写者未对其需修改内容未大括号括起来，上线故障应为填写者的责任。

    d、新增内容和修改内容要在第一行注释中写清楚

    B、nacos模板

    ​ 例如: 需要上线spring下增加dataresource.mysql和redis内容。

  redis:
    ##redis 单机环境配置
    host: {redis.url.env}
    port: 6379
    password: {redis.pwd.env}

  datasource:
    #t2
    # MySql
    url: jdbc:mysql://{mysql.url}:{mysql.port}/{bladex}{env}?useSSL=false&useUnicode=true&characterEncoding=utf-8&zeroDateTimeBehavior=convertToNull&transformedBitIsBoolean=true&tinyInt1isBit=false&allowMultiQueries=true&serverTimezone=GMT%2B8&allowPublicKeyRetrieval=true
    username: {mysql.user}
    password: {mysql.pwd}
