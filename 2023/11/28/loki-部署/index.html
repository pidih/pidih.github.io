    部署LOKI

    一、部署环境

    1、系统：Alibaba Cloud Linux2.19(centos)

    2、部署工具：使用docker-compose部署

    3、部署服务： loki, promtail, grafana, prometheus, alertmanager

    二、开始部署

    1、安装docker和docker-compose

    docker

    安装yum仓库管理工具
    yum install -y yum-utils

    下载阿里的docker-ce仓库

    yum-config-manager --add-repo https://mirrors.aliyun.com/docker-ce/linux/centos/docker-ce.repo

    如果是国外的机器，可直接添加docker官方源

    yum-config-manager --add-repo https://download.docker.com/linux/centos/docker-ce.repo

    安装和启动

    yum install docker-ce-20.10.9 -y
    docker-compose

    https://github.com/docker/compose/releases

    ring:   #配置哈希环,用于将日志数据分配给不同的Loki节点

        store: inmemory  #配置存储引擎为inmemory:即内存中存储
      replication_factor: 1   #配置复制因子为1，即每个节点只存储―份数据。
    final_sleep: 0s   #配置最终休眠时间为o秒，即关闭时立即停止。
  chunk_idle _period: 1h   #配置日志块的空闲时间为1小时。如果一个日志块在这段时间内没有收到新的日志数据，则会被刷新。
  max_chunk_age: 1h   #配置日志块的最大年龄为1小时。当一个日志块达到这个年龄时，所有的日志数据都会被刷新。
  chunk target_size: 2048576   #配置日志块的目标大小为2048576字节（约为1.5NB）·如果日志块的空闲时间或最大年龄先达到
  chunk_retain_period: 30s   #配置日志块的保留时间为3o秒。这个时间必须大于索引读取缓存的TL（默认为5分钟》。
  max_transfer_retries: 0   #配置日志块传输的最大重试次数为o，即禁用日志块传输。

schema_config:    #配置Loki的schema部分，用于管理索引和存储引擎。

storage_config:   #配置Loki的存储引擎相关的信息。

    cache_location:/tmp/loki/boltdb-shipper-cache   #配置BoltDB缓存文件的存储目录为/tmp/loki/boltdb-shipper-cache。

[root@bctest monitor]# cat prometheus/rule/nodealert.yaml   #这是告警配置
groups:
- name: UP
  rules:
  - alert: node
    expr: up{job="node_exporter"} == 0
    for: 1m
    labels:
      severity: crirical
    annotations:
      description: " {{ $labels.instance }} of job of {{ $labels.job }} has been down for more than 5 minutes."
      summary: "{{ $labels.instance }} down,up=={{ $value }} // up=0表示监控已断开连接，机器已断开连接"
  - alert: HostMemory Usage
    expr: (node_memory_MemTotal_bytes - (node_memory_MemFree_bytes + node_memory_Buffers_bytes + node_memory_Cached_bytes)) / node_memory_MemTotal_bytes * 100 >  85
    for: 1m
    labels:
      name: Memory
      severity: Warning
    annotations:
      summary: " {{ $labels.service}} 内存飙升 "
      description: "主机内存使用率超过80%."
      value: "{{ $value }}"
  - alert: HostCPU Usage
    expr: sum(avg without (cpu)(irate(node_cpu_seconds_total{mode!='idle'}[5m]))) by (instance,service) > 0.80
    for: 1m
    labels:
      name: CPU
      severity: Warning
    annotations:
      summary: " {{ $labels.service }} CPU飙升"
      description: "主机CPU使用率超过80%."
      value: "{{ $value }}"
  - alert: HostLoad 
    expr: node_load5 > 4
    for: 1m
    labels:
      name: Load
      severity: Warning
    annotations:
      summary: "{{ $labels.service }} 负荷飙升"
      description: " 主机负载5分钟超过4."
      value: "{{ $value }}"
  - alert: HostFilesystem Usage
    expr: 1-(node_filesystem_free_bytes / node_filesystem_size_bytes) >  0.8
    for: 1m
    labels:
      name: Disk
      severity: Warning
    annotations:
      summary: " {{ $labels.service }} 分区快满了"
      description: " 主机 [ {{ $labels.mountpoint }} ]分区使用超过80%."
      value: "{{ $value }}%"
  - alert: HostDiskio
    expr: irate(node_disk_writes_completed_total{job=~"Host"}[1m]) > 10
    for: 1m
    labels:
      name: Diskio
      severity: Warning
    annotations:
      summary: " {{ $labels.service }} IO读写过大"
      description: " 主机 [{{ $labels.device }}]磁盘1分钟平均写入IO负载较高."
      value: "{{ $value }}iops"
  - alert: Network_receive
    expr: irate(node_network_receive_bytes_total{device!~"lo|bond[0-9]|cbr[0-9]|veth.*|virbr.*|ovs-system"}[5m]) / 1048576  > 3 
    for: 1m
    labels:
      name: Network_receive
      severity: Warning
    annotations:
      summary: " {{ $labels.service }} 网络进来流量飙升"
      description: " 主机 [{{ $labels.device }}] 网卡5分钟平均接收流量超过3Mbps."
      value: "{{ $value }}3Mbps"
  - alert: Network_transmit

#!/bin/bash

    正则规则，

    如果需要调试正则可使用这两个网站 https://regex101.com/ https://regexr-cn.com/

    ‘(?P\d{4}-\d{2}-\d{2}\s\d\d:\d\d:\d\d.\d+)\s+(?P[A-Z]+)\s+(?P\d+)\s+(.?)(?:\d{4}-\d{2}-\d{2}\s*\d{2}:\d{2}:\d{2}.\d+|$)’

    把time当为参数，后面的规则匹配到的内容为参数值。后面用来替换timestamp.

    (?P<time>\d{4}-\d{2}-\d{2}\s*\d\d:\d\d:\d\d\.\d+)\s+   #匹配

2023-09-26 10:32:58.093

    意思是 level为参数，这是日志等级， 后面的规则匹配到的内容为参数值。

    (?P<level>[A-Z]+)\s+            #匹配
DEBUG 
INFO

    下面的意思是tread_id做为参数，匹配\d+ 数字N个，后面结空格或tab N个，匹配到的内容为参数值。

    (?P<tread_id>\d+)\s+        #匹配
27807

    后面所有的内容为最后匹配，**(.*? )*表示后面所有内容，(?:\d{4}-\d{2}-\d{2}\s\d{2}:\d{2}:\d{2}.\d+|$)直到遇到日期开头后结束匹配。

    (.*?)(?:\d{4}-\d{2}-\d{2}\s*\d{2}:\d{2}:\d{2}\.\d+|$)                #匹配
[http-nio-9510-exec-3] c.a.a.u.i.OperationLogInterceptor        178 : [GET]:/api/wms/tenant/info/owner,cost [4毫秒]

    接下来的是替换timestamp, 这个timestamp默认是采集日志时间为准，但有的时候你采集之前就有好几个月的日志，所以需要把采集时间用日志里面的时间替换掉。这样就算你今天采集的也可以使用日期查询到前几个月的日志，只有你采集目录下有那天的日志就可查到。

        - timestamp:       #这个意思就是替换timestamp的。
        source: time  #这行的意思是使用哪个源替换timestamp, 后面的值是time，就是之前匹配配置的参数。
        format: '2006-01-02 15:04:05.999'  #这行是格式化，我这边使用这个合适，其他多数使用RFC3339
        location: 'Asia/Shanghai'   #这行的意思是使用亚洲东八区的时区展示。

    不是docker启动就直接用命令，-server.http-listen-port 9999是监听 9999端口， –stdin可在前端调试，如果使用下面的命令就可直接进入调试模式， 然后直接把一串日志复制上去 回车，等待它处理。如果有none就说明那条没成功。如果有绿色标签出来说明成功了。 –config.file是指定配置文件，解析规则是按配置文件中的，所以这个不能少。

    promtail --stdin --dry-run --inspect -config.file=/etc/promtail/promtail-local-config.yaml -server.http-listen-port 9999

    当然也可以直接cat那日志让它解析,

    cat log.log |promtail --stdin --dry-run --inspect -config.file=/etc/promtail/promtail-local-config.yaml -server.http-listen-port 9999
